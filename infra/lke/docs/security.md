# Security Model

## Threat Model

This infrastructure runs for a 1-hour workshop and is destroyed immediately after. The threat model prioritizes:

1. **Student isolation** — one student cannot affect another's workspace
2. **Cluster integrity** — workspace pods cannot escalate privileges or access cluster resources
3. **Data exposure** — no sensitive data is stored; no persistent volumes

Non-goals (given the ephemeral nature):
- Long-term credential rotation
- Audit logging
- Network segmentation beyond namespace isolation

## Pod Security

Every workspace pod runs with a locked-down security context:

```yaml
securityContext:
  runAsNonRoot: true
  runAsUser: 1000
  allowPrivilegeEscalation: false
  capabilities:
    drop: ["ALL"]
```

- **Non-root**: Pods run as UID 1000 (the `coder` user)
- **No privilege escalation**: `allowPrivilegeEscalation: false` prevents setuid binaries
- **All capabilities dropped**: No Linux capabilities granted beyond baseline

## Access Control

- Each workspace gets a unique random password generated by `openssl rand -hex 4`
- Passwords are stored as Kubernetes Secrets and injected via environment variables
- Students receive passwords on printed access cards — no shared credentials
- No SSH access to nodes or pods — code-server over HTTPS only

## Network

- TLS terminated at the NodeBalancer (Linode-managed)
- Ingress routes paths (`/s01/`, `/s02/`, ...) to individual workspace services
- Workspace pods can reach:
  - `vllm:8000` (ClusterIP service for LLM inference)
  - External internet (for NBA API calls via nba-stats-mcp)
- No network policies enforced — acceptable for a 1-hour ephemeral cluster

## Cluster Lifecycle

- Provisioned T-24h before the workshop
- Destroyed T+30m after the workshop ends
- No persistent volumes — all data is ephemeral
- Terraform state stored locally (not in a remote backend)

## Credentials

| Credential | Storage | Rotation |
|-----------|---------|----------|
| Linode API token | Environment variable (`TF_VAR_token`) | Per-event |
| Workspace passwords | K8s Secrets | Generated per deployment |
| vLLM API key | Hardcoded `"not-needed"` (no auth) | N/A |

## Recommendations for Production Use

If adapting this infrastructure for longer-lived deployments:

1. Add NetworkPolicies to restrict pod-to-pod communication
2. Use a PodSecurityStandard (Restricted) at the namespace level
3. Enable audit logging on the LKE cluster
4. Store Terraform state in a remote backend with encryption
5. Add resource quotas to the workshop namespace
6. Use cert-manager for TLS instead of NodeBalancer termination
